## 休止中（フラグで再ON可能）

### BeReal型いいね受信枠（女性5件/日・ランダム解放）

- **状態**: 2026-05-28 に `LIKE_QUOTA_ENABLED` フラグで休止（β は OFF）。コード・テーブル・pg_cron は drop せず残置
- **再ON方法**: backend 環境変数 `LIKE_QUOTA_ENABLED=true` を設定するだけ。like_quota テーブルと pg_cron `generate-like-quota`（毎日 JST 0:00 に翌日枠生成）はそのまま稼働可能
- **判断トリガー**: (1) 女性ユーザーが「いいねを受けすぎて疲弊」のフィードバックを多数返した、(2) 男性ユーザーが「マッチが多すぎてやり取りが追えない」になった、(3) DAU が安定し露出スケジューリング（BeReal型）の射幸性が UX として効くと判断した時
- **実装コスト**: 低（既にコードと DB が残っている・フラグを True にして HomePage の受信枠カード文言を見直す程度）
- **期待効果**: 女性の精神的負荷分散、男性タイムラインの「今日見られる人」演出
- **リスク**: 復活時は HomePage の「ITEMS（男性向けいいねストック）」セクションとの UI 共存をどうするか要再検討（現状は男性向け在庫＝③ と女性向け受信枠＝① の両方が同時に出る設計のまま）

## 本番後・ユーザー数次第

### limited_admin ロール（制限付き管理者）

- **判断トリガー**: admin を自分以外に委譲/追加する必要が出たとき(運営チーム化・モデレーター追加)
- **実装コスト**: 中〜高（profiles.status または別 role カラム追加・require_admin を権限種別で分岐・admin 23本それぞれに limited 可否の線引き・フロント AdminGuard 分岐）
- **期待効果**: モデレーション業務を full admin 権限なしで委譲可能。limited_admin は BAN 可能(full admin と違い緊急停止対象にできる)
- **リスク**: RBAC 導入で認証認可の複雑度が上がる。セキュリティ修正と混ぜず独立 PR で設計すること
- **備考**: [2.5](2026-05-31)で「adminにBANチェック追加」を検討した際に派生したアイデア。当時は full admin=オーナー1人のため見送り

### 空きコマ機能

- **判断トリガー**: DAU 100超え
- **実装コスト**: 中（プロフィールに時間割追加・グリッド UI）
- **期待効果**: 阪大限定の強み活用、マッチ前の話題作り
- **リスク**: 時間割という個人情報の追加収集（PP 改訂が必要）

### アフタヌーン機能（12〜15時に女性が募集できる枠）

- **判断トリガー**: DAU 200超え（人数いないと機能しない）
- **実装コスト**: 中
- **期待効果**: 滞在時間延長、女性のアクティブ率向上
- **リスク**: 女性ユーザーが少ないと完全に機能しない

### プロフィール質問拡充（変な質問1〜2個・任意）

- **判断トリガー**: 本番リリース後
- **実装コスト**: 低
- **期待効果**: マッチ前のアイスブレイク
- **リスク**: 項目増加で回答放棄率上昇。1〜2個の任意項目に絞ること

### ミニゲーム

- **判断トリガー**: 滞在時間が KPI 課題になったとき
- **実装コスト**: 高
- **期待効果**: マッチ前後の滞在時間延長
- **リスク**: アプリの輪郭がぼやける、開発負担大

- **内容**: 長時間滞在で署名 URL(3600秒)が切れた際にプレースホルダ差し替え or 再フェッチ

## 却下（個人開発フェーズでは実装しない）

### 掲示板機能

- **却下理由**: 個人で24時間監視は不可、炎上リスク甚大
- AI 多段監視（Gemini Flash → GPT-4o-mini → Claude）の技術案はあるが、
  法人化または運営チーム複数人化が前提
- ただし **1対1チャットの通報後 AI 監視**は採用余地あり（要別途検討）
