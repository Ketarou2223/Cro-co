### BeReal型いいね受信枠（女性5件/日・ランダム解放）

- **再ON方法**: backend 環境変数 `LIKE_QUOTA_ENABLED=true` を設定するだけ。like_quota テーブルと pg_cron `generate-like-quota`（毎日 JST 0:00 に翌日枠生成）はそのまま稼働可能。**★再ON前に必須（2件）**: (1) `like.py:203-205` の `used_count` 更新を `SET used_count = used_count + 1 WHERE used_count < 上限` のアトミック更新に直すこと（現状は read-modify-write で並列リクエスト時に枠超過しうる。根拠: [6.4] 調査 2026-06-03）。(2) `like.py:153-163` の `should_count_quota` RPC 失敗時の扱いを再設計すること（現状は失敗時 `should_count=False` → 在庫消費なし・枠未カウントでいいね成立＝fail-open。quota 有効時はこれが枠/在庫の取りこぼしになる。単純 fail-close（いいね拒否）は RPC 瞬断で正規いいねを弾くため不可。リトライ or 事前取得キャッシュ等の安全な別戦略を検討。根拠: [10.6] 調査 2026-06-03）
- **判断トリガー**: (1) 女性ユーザーが「いいねを受けすぎて疲弊」のフィードバックを多数返した、(2) 男性ユーザーが「マッチが多すぎてやり取りが追えない」になった、(3) DAU が安定し露出スケジューリング（BeReal型）の射幸性が UX として効くと判断した時
- **実装コスト**: 低（既にコードと DB が残っている・フラグを True にして HomePage の受信枠カード文言を見直す程度）
- **期待効果**: 女性の精神的負荷分散、男性タイムラインの「今日見られる人」演出
- **リスク**: 復活時は HomePage の「ITEMS（男性向けいいねストック）」セクションとの UI 共存をどうするか要再検討（現状は男性向け在庫＝③ と女性向け受信枠＝① の両方が同時に出る設計のまま）

### Content-Security-Policy（CSP）の導入（XSS 多層防御の保険）

- **判断トリガー**: β後、または外部公開範囲を広げる時。[5.4] 調査で XSS 経路はゼロだが、多層防御として欠けている層を塞ぐタイミングで実施
- **実装コスト**: 中（React / Vite はビルド時にインライン初期化スクリプトを生成するため `'unsafe-inline'` を許可するか nonce/hash 方式が必要。Supabase・Google Fonts 等の外部オリジンを許可リスト化する設定作業あり。設定は Vercel `vercel.json` の `headers` が筋。導入後は全画面が壊れないか dev → prod で実機目視が必須）
- **期待効果**: 万一 XSS 経路が新たに生まれても悪性スクリプトの実行をブラウザ側で阻止する「最後の砦」になる
- **リスク**: CSP 設定を誤ると自アプリのスクリプトもブロックされ画面が動かなくなる → 独立 PR・実機確認必須。まず `Content-Security-Policy-Report-Only` ヘッダで違反ログを収集してから本適用する手順が安全
- **同時実施推奨**: 既存の `X-XSS-Protection: 1; mode=block`（`main.py:43`）は Chrome 78+ 削除済みで現代ブラウザでは実効なし。CSP 導入時に撤去または維持を判断する
- **GA4 導入に伴う申し送り**: GA4 を導入したため CSP 設定時に `*.googletagmanager.com` と `*.google-analytics.com` を `script-src` / `connect-src` に追加する必要あり（2026-06-04 追記）
- **関連**: [5.4]（2026-06-03 調査・XSS 穴ゼロ確認）

### 通報の重複抑止（UNIQUE 制約 + アプリ層 409）

- **判断トリガー**: β後、または同一通報者が同一相手を繰り返し通報し管理画面ノイズが問題化したとき
- **実装コスト**: 低（migration 1本 + safety.py に重複チェック or upsert を追加）
- **期待効果**: 管理ダッシュボードの通報一覧に同一ペアの重複エントリが積み重ならなくなる
- **リスク**: なし（ユーザー体験への影響なし。通報操作の重複ブロックはサイレントで返してよい）
- **現状の安全性**: rate limit 10/min（ユーザー単位・JWT sub キー）で機能しており、重複通報が安全インシデントになることはない。β 受容範囲
- **具体的修正方針**: (1) migration: `reports` テーブルに `UNIQUE(reporter_id, reported_id)` 制約追加（冪等・既存重複行があれば先に除去）。(2) `safety.py:report_user` で INSERT を upsert または重複チェック後スキップに変更（409 または 204 で返す）
- **関連**: [8.5]（2026-06-03 調査・β受容確認）

### メアド正規化による重複登録防止（`+` エイリアス・大文字小文字対策）

- **判断トリガー**: ブロックすり抜け被害の報告が発生した、または阪大メールが将来 `+` 配送に対応した、または本番リリース前のセキュリティ強化フェーズに入ったとき
- **実装コスト**: 低〜中（DB トリガー1本の改修・migration 追加・prod 手動適用）
- **期待効果**: 同一人物が `+alias` で複数アカウントを量産しブロックをすり抜けることを根本から防ぐ
- **リスク**: トリガー改修は prod に直接影響するため独立 PR + dev 先行確認が必須
- **選択肢A（推奨・最小変更）**: 既存 BEFORE INSERT トリガー `enforce_university_email_domain`（migration 034）に `+` 以降除去・lowercase 化を追加し、`auth.users` に正規化後メールが存在しないか確認。[5.1] の砦に相乗りできるため追加コストが最小。prod 適用はオーナー SQL Editor 手動。
- **選択肢B（参考）**: `profiles` テーブルに `canonical_email text UNIQUE` カラムを追加し `handle_new_user`（AFTER INSERT）で書き込む。ただし AFTER INSERT のため auth.users が先に作成されており、重複時に auth.users の孤立行が残る原子性問題がある。
- **選択肢C（大改修・将来）**: サインアップフローを FastAPI 経由に変更し、backend で canonical email の重複チェックを実施。`CLAUDE.md §5` の `lib/supabase.ts` と `AuthContext.tsx` に影響するため独立大型 PR。
- **関連**: [5.2]（2026-06-03 調査・🟡クローズ）/ migration 034 `enforce_university_email_domain`

## 本番後・ユーザー数次第

### ブロック時のデータ物理削除（会話・いいね・足跡・マッチ）

- **方針（確定・プライバシー優先）**: 身内向けゆえ、ブロック時は当該相手との会話/メッセージ/いいね（双方向）/足跡/マッチを物理削除する。退会時は現状の CASCADE 全消しで既に達成済み（[4.2] 2026-06-02 確認）。
- **判断トリガー**: β後。「ブロック＝隠す（現実装）」で身内人数のうちは実用上足りるため、β は現状維持で出す。
- **実装コスト**: 中〜高（ブロック時に複数テーブルを物理削除する新規処理。RLS・FK CASCADE 整合の確認必須）
- **期待効果**: ブロック後も相手の画面に「退会済み」表示が残らない（完全消滅）。相手に「ブロックされた」と気付かれにくい（退会と区別不能な挙動になる）
- **リスク**: 物理削除は不可逆。誤爆すると相手の会話まで消える。独立 PR で設計＋テスト必須
- **設計上の必須要件（実装時に必ず守る）**:
  1. `blocks` レコード自体は残す（解除不可仕様・再マッチ防止のため。会話等だけ消し、ブロック事実は保持）
  2. 両者の視点から消す（ブロックした側・された側どちらの画面からもマッチ/会話が消滅）。された側には理由を示さない＝「相手が退会した」と区別不能でよい（ブロックは静かに、の原則）
  3. 物理削除のトランザクション整合（messages dangling・マッチ片側残り等を作らない）
  4. この変更で `match.py:108` の `is_deleted` 匿名表示分岐と `privacy_purge.py:81` の `purge_deleted_user_messages()` の dead code の去就も同時に決める（機能させる or 削除）
- **関連**: [4.2]（退会 CASCADE 挙動の正式仕様化）/ [3.2]（ブロック解除不可の設計）/ HANDOFF §5 dead code リスト
- **is_deleted 除去は本機能と一括で**: `match.py` / `schemas/match.py` の `is_deleted` 分岐はソフトデリート廃止後バックエンド到達経路ゼロだが、フロント（ChatPage/MatchesPage/LikesReceivedPage/lib/db.ts）が UI で参照中。除去はこの物理削除機能の設計時にフロント・バック一括で実施（単独削除は型不整合を招くため不可）。

### limited_admin ロール（制限付き管理者）

- **判断トリガー**: admin を自分以外に委譲/追加する必要が出たとき(運営チーム化・モデレーター追加)
- **実装コスト**: 中〜高（profiles.status または別 role カラム追加・require_admin を権限種別で分岐・admin 23本それぞれに limited 可否の線引き・フロント AdminGuard 分岐）
- **期待効果**: モデレーション業務を full admin 権限なしで委譲可能。limited_admin は BAN 可能(full admin と違い緊急停止対象にできる)
- **リスク**: RBAC 導入で認証認可の複雑度が上がる。セキュリティ修正と混ぜず独立 PR で設計すること
- **備考**: [2.5](2026-05-31)で「adminにBANチェック追加」を検討した際に派生したアイデア。当時は full admin=オーナー1人のため見送り
- **セキュリティ連動メモ（2026-06-03 [9.4]）**: 運営を複数人化する際は `admin_logs` の真の append-only 化（`service_role` からも UPDATE/DELETE を REVOKE・トリガー保護）を同時に設計すること。現状は「単一信頼主体（オーナー）」前提で Studio 直削除を許容しているが、複数 admin になると前提が崩れる

  法人化または運営チーム複数人化が前提
