## 7. リリース前セキュリティチェックリスト

UTopia 事案（2026年4月）を教訓とし、本番リリース前にあらゆる手段でセキュリティを検証する。

### 実施時期
- β版リリース 2週間前 / 本番リリース 1ヶ月前

### 1. AI レビュー（複数 AI で交差検証）
- [ ] Claude Opus / Gemini Pro / GPT-5 による全コードレビュー
- [ ] 3つの AI で指摘が一致した箇所を優先対応

### 2. Deep Research
- [ ] 「マッチングアプリ 脆弱性 事例」「Supabase セキュリティ 落とし穴」で Web 調査
- [ ] OWASP Mobile Top 10 / OWASP API Security Top 10 を全項目チェック

### 3. 自動スキャン
- [ ] Snyk / Trivy で依存関係の脆弱性スキャン
- [ ] semgrep で SAST / OWASP ZAP で DAST
- [ ] GitGuardian で commit 履歴のシークレット漏洩チェック

### 4. 手動ペネトレーションテスト
- [ ] 認証バイパス: フロントを通さず curl で各 API 直叩き
- [ ] 認可 / IDOR: 他人の user_id・連番アクセス
- [ ] SQL injection: 全入力に `' OR '1'='1`
- [ ] XSS: 全テキスト入力に `<script>alert(1)</script>`
- [ ] CSRF: 別オリジンからの POST/PATCH/DELETE
- [ ] レースコンディション: 並列リクエストで二重マッチ等
- [ ] 大量データ攻撃: 配列に1000要素、文字列に100KB

### 5. プライバシー検証
- [ ] 学生証画像が承認後3日で完全削除されているか
- [ ] 本名・学籍番号がハッシュ化されているか
- [ ] 退会時に全データが論理削除されるか
- [ ] エクスポート機能で本人が自分のデータを取得できるか

### 6. 法的チェック
- [ ] インターネット異性紹介事業届出（大阪府公安委員会）
- [ ] PP が最新法令に準拠 / 利用規約に必要条項
- [ ] 18歳未満排除（年齢確認）が実装されている
- [ ] 個人情報保護委員会への漏洩時通報手順を文書化

### 7. インフラ検証
- [ ] Supabase Storage 全バケットの Public/Private 設定
- [ ] 全テーブルの RLS 有効化と適切なポリシー
- [ ] service_role キーがフロントに混入していない
- [ ] 環境変数が git に追跡されていない
- [ ] Vercel・Render のシークレットが正しく分離されている

### 8. 第三者監査
- [ ] 信頼できるエンジニアに有償でコードレビュー依頼
- [ ] 大学のセキュリティ研究室への監査依頼を検討

### UTopia 事案の教訓（参考）
2026年4月にリリースされ1週間で事実上終了したマッチングアプリ。バイブコーディング製コードに以下の致命的脆弱性があった:
1. 認証なしで全ユーザーデータを JSON 取得可能
2. マイナンバー画像が公開ストレージに保管
3. メールドメイン制限がクライアント側のみ
4. ユーザー側で認証ステータスを書き換え可能
5. 課金アイテムが int max まで購入可能（上限チェックなし）
6. service_role キーがフロント JS に埋め込み
7. インターネット異性紹介事業届出番号がサイト未掲載

Cro-co では 2026年5月時点の監査で 1〜6 は対策済みを確認。7 は β 前に対応予定。
